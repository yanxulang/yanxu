# 模块限定类型与跨模块对象模型

本文记录 Issue #14 的实现约束。目标不是把点号拼进类型名，而是让静态检查器、树解释器、字节码 VM 和语义工具共享同一套模块与类型身份语义。

## 语法与 AST

所有命名类型使用统一的 `TypePath`：

```rust
TypePath {
    segments: Vec<TypePathSegment>,
    span: Span,
}

TypePathSegment {
    name: String,
    span: Span,
}
```

`视图`、`基础.视图`和`包.控件.按钮`分别是一个、两个和三个路径段。每段保留自己的跨度，路径同时保留完整跨度；诊断可以标出失败的中间段或最终段。`Display`只输出规范点号形式。

`TypeKind::Named`和泛型基础类型持有 `TypePath`。类声明的 `superclass`和`protocols`也持有 `TypePath`，不再使用 `String`。联合、可空、泛型、元组、任务和函数类型继续递归包含同一 `TypeKind`，因此限定路径自动覆盖所有类型位置。裸本地类型只是单段 `TypePath`，不建立第二套规则。

`公 引「path」为 名；`是顶层声明。普通`引`只建立当前模块私有绑定；`公 引`还把模块绑定放入导出表。`公 引`出现在块内时由解析器直接拒绝。

## 规范模块与类型身份

规范身份与源码别名分离：

```rust
enum ModuleId {
    Standard { name: String },
    Package { package: String, module: String },
    Archive { module: String },
    File { canonical_path: String },
    Memory { name: String },
}

enum TypeDeclarationKind { Class, Protocol }

struct TypeId {
    module: ModuleId,
    name: String,
    kind: TypeDeclarationKind,
}
```

包编译优先使用锁图中的稳定包 ID 与包内相对路径；当前包使用包名、版本和相对包根路径。标准模块进入独立命名空间。YXB 使用已有的 `app:` / `pkg:` 逻辑模块 ID，避免把构建机绝对路径写入制品。没有清单的源码执行才退化为规范文件路径；内存源码使用显式的内存模块名。

导入别名只用于解析访问路径。`基础.视图`和`核心.视图`若指向同一模块，最终都链接到同一个 `TypeId`。同名声明只要 `ModuleId`不同就保持不同身份。

## 模块摘要与名称解析

检查器使用显式模块摘要：

```rust
ModuleSummary {
    module_id: ModuleId,
    exports: BTreeMap<String, ExportedSymbol>,
}
```

`ExportedSymbol`区分值、函数、类、协议和重导出模块。类摘要保存 `TypeId`、构造参数、公开实例/静态成员、完整成员签名、可见性、只读属性、父类和协议身份；协议摘要保存必需域与方法签名。摘要中的嵌套类型引用使用解析后的规范身份。

每个正在检查的模块拥有独立的本地类型表和导入别名表。进程级注册表只按 `TypeId`保存类形状、协议形状及关系图。离开导入模块后，其裸名称不会进入消费模块作用域；消费模块只能通过导入别名或公开重导出路径访问它。

所有类型位置调用同一个路径解析入口。解析顺序为：

1. 单段内建类型；
2. 当前模块的本地类或协议；
3. 首段对应的导入模块；
4. 逐段跟随公开重导出模块；
5. 最终段必须是当前上下文要求的类或协议。

中间段不是模块、最终成员不存在、成员未公开或声明种类错误分别产生稳定诊断。公共声明的签名还会检查其所有规范类型是否能由模块消费者访问，防止泄漏本地私有类型。

## 解释器链接

模块对象携带 `ModuleId`，类和协议对象携带 `TypeId`。执行类声明时，单段父类/协议从当前环境解析，多段路径逐级读取模块公开导出；不会把导入类型复制到本地环境。

实例的 `是`、继承关系和协议符合性比较 `TypeId`。类的显示名仍可使用短名称，但诊断和调试信息在有歧义时显示限定身份。方法继续保存其声明模块闭包与目录；跨模块继承只连接父类对象，不改写方法闭包，因此动态派发、`父.方法`、父类初始化和异常栈沿用同模块语义。

公开导入本身是普通模块对象的导出引用。重导出不会创建模块副本或新的类型身份，任意深度路径通过相同模块属性访问递归工作。

## 字节码与 VM 链接

字节码格式提升到 2。`Chunk`保存声明模块的 `ModuleId`；`ClassPrototype`和`ProtocolPrototype`保存规范 `TypeId`。参数、返回值、域、父类、协议与 `Instruction::IsType`使用结构化运行时类型引用，不再保存一个需要反复切割的类型字符串。

运行时类型引用包含结构化源路径以及在编译时可确定的规范目标。编译时不能确定的导入或重导出目标作为链接引用保留；VM 装载模块后通过模块导出解析，并以被解析类/协议对象的 `TypeId`完成检查。源代码别名因此只是链接位置，不是永久身份。

应用编译器把每个块的模块身份设为既有 `app:` / `pkg:`逻辑 ID，并与导入改写一同递归处理嵌套函数和方法。反序列化严格拒绝格式 1 和损坏的模块/类型身份，不把旧裸字符串解释为新身份。YXB 外层格式仍为 1，但其 `bytecode_format`必须为 2。

VM 类、协议、方法所有者、`父`查找和协议集合全部使用 `TypeId`。模块对象携带 `ModuleId`并导出公开重导出模块，行为与树解释器一致。

## 公开重导出、冲突与循环

`公 引`导出导入别名对应的模块命名空间。重导出路径保留原声明的 `TypeId`及原始模块，同时摘要记录当前暴露路径。模块内重复声明或重导出同名由现有声明冲突检查拒绝；不会依赖 `HashMap`覆盖顺序。

静态检查、解释器、VM 和 YXB 编译都使用有序装载栈。再次进入栈中模块时报告完整确定链。重导出循环属于模块循环，不设静默占位摘要。

## 文档、API 与 LSP

Markdown 以完整路径显示限定类型，类和协议链接按规范所有者定位；目录文档为公开重导出模块建立索引，并把 facade 暴露路径链接到原始声明。

API manifest 格式提升到 2。每个类型声明至少输出 `owner_module`、`qualified_name`和声明种类；类型引用、父类及协议为结构化对象。重导出同时记录 `exposed_path`和原始模块身份。格式 1 消费者会看到明确版本变化，迁移说明记录字段映射。

LSP 解析当前文档的导入图但不执行模块。`.`补全读取公开摘要；限定类型 hover 显示规范名称、所有者、父类、协议和公开成员；definition 跟随重导出链到原始声明。导入别名 rename 只改同一文档中解析到该绑定的路径段。当前语义索引不能证明反向依赖图完整，因此外部限定类型的跨文件 rename 明确返回不可用；本模块类型仍按本地符号 ID 安全改名。实现不进行任何基于短名称的跨文件文本替换。

## 兼容性

同模块裸类、协议、现有内建类型、普通模块值访问和类构造保持兼容。唯一有意移除的行为是：检查某个导入模块后，其类或协议短名称偶然进入消费模块全局注册表。此类源码会得到要求使用`模块.类型`的明确诊断。

树解释器与 VM 都以同一规范身份完成运行时边界检查。格式化器只通过 `TypePath::Display`输出类型路径，保证点号无空格、递归类型稳定且 parse → format → parse 等价。
