# 言序 1.x 兼容与演进政策

言序从 1.0.0 起遵循 SemVer。此承诺覆盖语言源码、标准库公开成员、包与锁格式、CLI 稳定命令、Rust 嵌入 API、C ABI schema、JSON schema 与 Tree-sitter 节点类型。

## 版本边界

- **补丁版本**只能修复实现偏离规范、诊断位置、崩溃、安全和不改变成功程序结果的工具问题。
- **次版本**可以增加向后兼容的语法、标准库成员、可选字段和工具命令；既有 1.x 成功程序必须继续成功并保持规定的可观察结果。
- **主版本**才可删除公开能力、改变既有语义、收紧规范允许的成功输入，或改变公共格式主版本。

未记录的内部 Rust 模块、VM 指令布局细节、性能数值、诊断的非关键措辞和未规定行为不属于稳定 API。任何安全修复都可以立即拒绝此前会导致越权、内存/资源失控或归档欺骗的输入，并必须进入安全说明和变更记录。

## 解析资源边界

解析器对语句、表达式和类型结构统一实施 32 层深度上限，超限时返回确定性的语法错误，不进入格式化、语义分析或执行阶段。该边界同时覆盖括号与集合嵌套、一元与二元表达式链、成员和下标链、嵌套泛型及控制流块。

此限制只收紧此前可能耗尽进程栈的无界输入。超过边界的源码应把长表达式拆成多个具名中间值，把深层控制流抽成独立函数，并用具名类或协议拆分嵌套类型；边界内源码的语义、格式和执行结果不变。

## 弃用周期

公开能力移除前必须：

1. 登记稳定诊断编号、替代形式、引入与预计移除版本；
2. 至少跨越一个 1.x 次版本，并保留不少于六个月；
3. 提供`yanxu 迁`自动迁移，或解释无法自动迁移的原因；
4. 在变更记录、文档和兼容语料中验证迁移。

1.x 内不得移除已弃用源码能力；最早只能在 2.0 移除。警告默认不使构建失败，自动化可使用`yanxu 迁 --检查`将其作为非零退出。

## 提案流程

语言、标准库或公共格式变化先提交 YXP（Yanxu Proposal）。提案必须包含动机、完整语法/类型/运行语义、错误模型、权限与资源影响、双执行器方案、工具链影响、兼容与迁移策略、测试和拒绝的替代方案。流程为草案、试验、接受/拒绝、稳定；接受不等同于立即发布。

模板和编号规则见[`proposals/README.md`](proposals/README.md)。破坏性提案只有在新主版本路线获批后才能接受。

## 验证门槛

所有 1.x 发行必须通过`compat/`历史语料、双执行器一致性、资源边界、严格 Clippy、WASI、JSON schema、Cargo 发布包、文档/官网/VS Code 生产构建。公开格式规则见[`spec/language/v1/formats.md`](spec/language/v1/formats.md)。

CI 还会以不可变的`v1.1.15`标签为结构化基线，比较语言关键字与语法、Tree-sitter 节点、标准库 API、清单与锁文件字段、YXB/字节码格式、原生与嵌入 ABI、工程协议、稳定 CLI 和全部 JSON Schema。删除或改变既有契约要求主版本，纯新增公开契约要求次版本，公开面不变才允许补丁版本；浅克隆或缺失基线标签不能通过发布门禁。
