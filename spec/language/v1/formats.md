# 公共格式

公共格式均以独立整数主版本标识。实现必须拒绝未知主版本，不得猜测解析。

| 格式 | 当前版本 | 兼容读取 | 标识 |
| --- | ---: | --- | --- |
| 包清单 | 2 | 1、2 | `[包].格式` |
| 锁文件 | 2 | 1、2 | `lock_version` |
| 字节码块 | 1 | 1 | `Chunk.format_version` |
| YXB 应用 | 1 | 1 | `format_version`与内容校验和 |
| 工程协议 | 1 | 1 | `protocol_version` |
| 原生扩展 ABI | 1 | 1 | `abi_version`与`struct_size` |
| 测试报告 | 1 | 1 | `schema_version`和`test-report-v1` URI |
| 兼容报告 | 1 | 1 | `schema_version`和`compatibility-report-v1` URI |
| C ABI 结果 | 1 | 1 | JSON `schema`字段 |
| 标准库清单 | 1 | 1 | `schema_version`和`stdlib-api-v1` URI |

同一主版本中可以增加接收方明确忽略的可选字段；删除字段、改变字段类型/语义/默认值都需要新主版本。锁文件、字节码、YXB、工程协议和原生 ABI 要求精确主版本匹配；报告消费者应按 schema URI 选择解析器。

## 清单与锁文件格式 2

`yanxu-package`是`言序.toml`和`言序.lock`的唯一规范实现。纯言序编写的独立工程工具`yanbao`只通过版本化 JSON 协议调用它，不实现第二套 TOML、SemVer、依赖选择或锁定语义。清单兼容言序惯用的中文裸键和标准 TOML 引号键。

格式 2 清单的最小必需表为`包`；`依赖`和`权限`按需增加。可选的`开发依赖`、`导出`、`资源`、`构建`、`工作区`和`原生`表不得包含任意脚本或安装钩子。依赖可以使用与真实包名不同的别名；模块路径`包:别名/导出`必须通过目标包的显式导出表。

格式 2 锁文件必须包含：

- 顶层普通与开发依赖别名到稳定包 ID 的边；
- 每个直接与传递包的名称、精确版本、来源、Git 修订和内容校验和；
- 每个包的依赖边、入口、导出表、目标平台、最低言序版本和选中的原生制品；
- 顶层清单校验和、生成器版本和确定性排序。

运行、检查和构建必须使用这份完整图，不得在导入时回退到依赖自己的清单临时选版。顶层源码只能导入顶层清单显式声明的依赖；传递包只能沿自己在锁图中的边导入子依赖。

Git 清单可以保存分支、标签或精确提交，锁文件只保存解析后的精确提交。显式更新非精确修订时必须查询远端；离线安装只能复用已缓存或`vendor`辖制、且与锁文件逐包一致的内容。

已通过清单、版本、精确修订和内容校验的包根可以自动用于模块加载；这项授权不得合并依赖清单的文件、网络、监听、环境、进程或原生扩展权限。

## 安全制品与确定性

远程索引`tar.gz`和 YXP 制品只可包含相对路径的普通文件和目录。解压器必须拒绝父目录/绝对路径、符号链接、硬链接、块/字符设备、管道和重复写入，并在写入前检查压缩体积、条目数、路径长度、单文件和展开总量上限。

路径依赖按相对路径排序，将路径、文件长度和文件字节写入 SHA-256。`.git`、`.yanxu`、`target`、`build`、`.DS_Store`和任意层级的`言序.lock`属于工作区或派生制品，不参与内容校验；其他文件变化必须使旧锁定失效。YXB/YXP 使用固定顺序和元数据，相同输入必须生成逐字节相同的制品。

YXB 的权限字段只表达应用申请。宿主运行时必须把自身授权视为不可突破的上限，逐项取宿主权限与应用申请的交集；YXB 不得自行提升文件、网络外连、TCP 监听、UDP 绑定、环境、进程或原生扩展权限。嵌入 API 与 CLI/VM 必须复用同一合并语义。

外部 YXB 在整体读取前须检查普通文件类型和压缩体积，并以限长方式解压。格式 1 的当前预算为：压缩文件 64 MiB、解压 JSON 256 MiB、4096 模块、单模块序列化 32 MiB、100 万条指令、10 万函数、16384 类、100 万调试位置、4096 资源、单资源 16 MiB、资源总量 128 MiB。实现还必须拒绝重复 JSON 对象键、非法/重复模块与资源索引、不存在的入口、归档外导入、危险资源路径、资源摘要不符和不兼容字节码。

YXB 的内容 SHA-256 是完整性检查，不是发布者数字签名。攻击者若能改写制品，也能重新计算普通摘要；消费者不得据此声称发布者身份已验证。可选签名必须通过后续格式提案定义签名范围、算法、密钥标识、轮换、撤销与信任策略，不能在格式 1 中暗加不兼容语义。
