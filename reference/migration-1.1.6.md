# 从言序 1.1.5 迁移到 1.1.6

1.1.6 是稳定性、安全和发布一致性修复版本，不改变语言规范 1 的语法，也不要求应用重写业务源码。升级重点是重新构建 YXB/standalone、重新生成锁文件，并确认宿主授权符合更严格的权限上限。

## YXB 权限模型

YXB 内的权限是应用“申请”的能力，不是最终授权。运行时有效权限始终是：

```text
有效权限 = 宿主权限上限 ∩ YXB 申请权限
```

因此，受限嵌入宿主运行申请了文件、网络、监听、环境、进程或原生扩展权限的 YXB 时，应用仍不能突破宿主上限。嵌入方可通过公共`PermissionSet::intersect`预览或复用同一合并语义；CLI 与 VM 执行 YXB 时会自动执行合并。

1.1.6 还会在读取前限制 YXB 压缩文件大小，并校验解压后 JSON、模块/指令/函数/类/调试信息数量、资源数量和大小、模块 ID、内部导入、资源路径与摘要。旧版未压缩 YXB v1 继续兼容，但建议用 1.1.6 重建制品，以写入运行时版本、目标、构建模式和提交 SHA。

## 摘要不等于签名

YXB 的 SHA-256 内容校验只能发现意外损坏或未同步篡改。能够修改 YXB 的人也能重新计算摘要，因此摘要不能证明发布者身份、组织授权或供应链来源。1.1.6 不把摘要显示为“已签名”。可选发布者数字签名、密钥轮换和信任策略留给后续 YXB 格式提案；在此之前应通过 HTTPS Release、受保护标签、Release 校验文件和组织审核验证来源。

## 原生扩展信任边界

原生动态库是进程内受信任代码，不受言序语言沙箱完全约束。1.1.6 会限长读取、校验目标和 SHA-256，把已验证字节复制到进程私有的内容寻址只读路径并再次校验后装载，同时保证 ABI 异常输出被释放；这些措施防止常见替换竞态和资源泄漏，但不能约束动态库调用任意操作系统 API。只启用经过独立审核的原生包。

## 包与锁文件

清单、锁文件和 vendor 清单使用原子替换；同一项目的编辑、安装和更新由跨进程项目锁串行化。更新解析失败时保留上一份完整锁文件，不再先删除旧锁。升级后执行：

```sh
yanbao install
yanbao check
yanbao test
yanbao build --release
```

言包从 1.1.6 起采用独立 Release：发布包内含由言序源码预编译的 standalone，不要求目标机器预先安装言序；从源码参与言包开发时仍使用匹配版本的`yanxu`。

## CI 与发布

`v1.1.6`只能精确指向已通过可复用完整质量门禁的`origin/main`提交。核心运行时和`yanxu-package`拥有不同标签与 Release，不能把共享包 crate 混入运行时二进制 Release，也不能让同一版本号对应多个代码快照。
